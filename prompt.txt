# Build “FareProof” — A Private Chrome Extension That Captures ITA Matrix Fares and Verifies Whether They Are Actually Bookable

You are a senior browser-extension architect, TypeScript engineer, airfare-pricing systems designer, reliability engineer, and product designer.

Build a production-quality private Chrome extension called **FareProof**.

I will install it manually in my own Chrome browser using **Load unpacked**. It is not initially intended for publication in the Chrome Web Store.

The extension’s purpose is to:

1. Observe favorable flight results I find in ITA Matrix.
2. Capture the complete itinerary and fare construction.
3. Search or open BookWithMatrix and other supported airline or travel-booking sites.
4. determine whether the same itinerary is offered at approximately the same all-in price;
5. distinguish between:

   * an itinerary merely displayed by ITA;
   * a fare found by another search engine;
   * a fare that reaches a retailer’s checkout flow;
   * a fare that appears genuinely ticketable;
6. notify me immediately when a sufficiently close, credible, bookable match is detected;
7. preserve enough evidence that I can quickly complete the booking myself.

The extension must not purchase anything, submit payment, enter passenger details, bypass a CAPTCHA, defeat bot protection, impersonate another user, circumvent access controls, or violate a website restriction.

It is a **research, verification, and notification assistant**, not an autonomous purchasing bot.

---

The primary user workflow should be:

1. I run a normal ITA Matrix search.

2. I open or select a favorable result.

3. FareProof detects the itinerary.

4. A compact button appears:

   **Watch and verify this fare**

5. FareProof captures:

   * origin and destination;
   * travel date;
   * departure and arrival times;
   * marketing carrier;
   * operating carrier;
   * marketing flight number;
   * operating flight number;
   * cabin;
   * booking class;
   * fare basis;
   * validating carrier, when available;
   * point of sale;
   * currency;
   * base fare;
   * taxes;
   * carrier-imposed surcharges;
   * all-in price;
   * number of passengers;
   * trip type;
   * stops;
   * aircraft;
   * fare rules and warnings;
   * raw ITA JSON, when available;
   * relevant visible page text as a fallback;
   * capture timestamp;
   * source URL;
   * a normalized itinerary fingerprint.

6. FareProof creates a **fare watch**.

7. It attempts verification through supported channels.

8. It classifies each result by confidence.

9. It alerts me when a result meets the configured match criteria.

10. The notification gives me one-click access to:

    * the matching retailer;

The experience should feel like an intelligent airfare forensic tool, not a generic price tracker.

# 2. Important airfare concepts

Do not treat two flights as identical solely because they use the same aircraft and depart at the same time.

The system must understand the distinction between:

* marketing carrier;
* operating carrier;
* marketing flight number;
* operating flight number;
* validating carrier;
* booking class;
* cabin;
* fare basis;
* filed fare;
* codeshare inventory;
* point of sale;
* currency;
* tax and surcharge composition;
* passenger count;
* ticketing channel;
* branded fare;
* itinerary-level price;
* segment-level availability.

Example:

* WestJet-marketed flight: `WS 5943`
* operated by Condor: `DE 2455`
* booking class: `D`
* cabin: business
* fare basis: `DZ0D0HNS`
* validating or fare-owning carrier: WestJet
* ITA price: CAD 1,313.67

A retailer showing Condor `DE 2455` for CAD 2,500 is not an exact fare match merely because it is the same physical flight.

FareProof must preserve and compare both identities.

# 3. Core match philosophy

A price alert must not be based on price alone.

A match must evaluate:

## 3.1 Itinerary identity

Compare:

* origin airport;
* destination airport;
* travel date;
* local departure time;
* local arrival time;
* stop count;
* connection airports;
* marketing carrier;
* marketing flight number;
* operating carrier;
* operating flight number;
* segment sequence;
* cabin for every segment.

## 3.2 Fare identity

Compare, when available:

* booking class;
* fare basis;
* validating carrier;
* fare owner;
* branded fare;
* point of sale;
* passenger type;
* one-way versus round-trip construction;
* change and cancellation conditions;
* baggage inclusion.

## 3.3 Price identity

Compare:

* original currency;
* total price;
* base fare;
* taxes;
* surcharges;
* passenger count;
* whether the displayed amount is per passenger or for the entire booking;
* whether the retailer adds a booking fee;
* whether currency conversion has occurred;
* whether the price includes mandatory fees.

## 3.4 Bookability evidence

Classify evidence as:

1. **ITA only**
2. **Search result reproduced**
3. **Retailer result reproduced**
4. **Exact itinerary selectable**
5. **Fare survives selection**
6. **Passenger-details page reached**
7. **Payment page reached without entering payment**
8. **Manual confirmation required**
9. **Unavailable or failed repricing**

The extension must never claim that a fare is definitively ticketed or guaranteed unless an actual ticket exists, which this extension will not create.

Use language such as:

* “Exact price reproduced”
* “Likely bookable”
* “Reached passenger-details stage”
* “Retailer repriced”
* “Search-only match”
* “Possible codeshare mismatch”
* “Manual verification needed”

Never display “Guaranteed bookable.”

# 7. ITA Matrix capture

Build the ITA adapter first.

The adapter should detect:

* search-results pages;
* itinerary details;
* fare construction;
* fare rules;
* selected result;
* any BookWithMatrix-compatible JSON;
* structured state exposed in page scripts or browser-visible network responses.

Do not depend exclusively on one extraction technique.

Implement three capture strategies.

## Strategy A: Embedded structured-data extraction

Inspect script blocks and page state for structured itinerary data.

Parse only data already delivered to the user’s browser.

Validate all extracted payloads using Zod.

## Strategy B: DOM extraction

Read the selected itinerary and fare details from the rendered page.

Use a `MutationObserver` to detect:

* result selection;
* result expansion;
* fare-detail opening;
* navigation between result pages;
* price changes.

## Strategy C: Manual JSON import

Add a side-panel action:

**Paste ITA or BookWithMatrix JSON**

Allow the user to paste JSON like:

```json
{
  "ext": {
    "totalPrice": "CAD1313.67"
  },
  "itinerary": {
    "...": "..."
  }
}
```

Parse and normalize it into the same domain object used by automatic extraction.

The manual import path is essential because it provides a reliable fallback when ITA changes its page structure.

# 9. Canonical itinerary fingerprint

Generate multiple fingerprints rather than one overly rigid key.

## Physical-flight fingerprint

Based on:

* origin;
* destination;
* date;
* local departure time within a small tolerance;
* operating carrier;
* operating flight number.

## Marketing-flight fingerprint

Based on:

* origin;
* destination;
* date;
* marketing carrier;
* marketing flight number;
* cabin.

## Fare fingerprint

Based on:

* marketing-flight fingerprint;
* booking class;
* fare basis;
* validating carrier;
* currency;
* passenger composition;
* point of sale.

## Journey fingerprint

Based on the ordered sequence of all physical segments.

Store all fingerprints on the watch and each observation.

This lets the system detect cases such as:

* exact WestJet-marketed match;
* same physical Condor flight but different marketing carrier;
* same itinerary but different booking class;
* same fare basis with a small price difference;
* same price but incorrect cabin;
* same outbound but missing or different connecting segments.

# 11. Comparison engine

Build a transparent, explainable comparison engine.

Do not return only `true` or `false`.

Return:

```ts
interface ComparisonResult {
  overallClassification:
    | "exact"
    | "strong"
    | "possible"
    | "mismatch"
    | "insufficient-evidence";

  score: number;

  priceComparison: PriceComparison;
  itineraryComparison: ItineraryComparison;
  fareComparison: FareIdentityComparison;
  verificationComparison: VerificationComparison;

  matchedFields: MatchField[];
  mismatchedFields: MatchField[];
  unknownFields: MatchField[];

  humanSummary: string[];
  alertEligible: boolean;
}
```

Example explanation:

```text
STRONG MATCH — 91%

✓ Same physical flight: DE 2455
✓ Same date and departure time
✓ Business cabin
✓ Total CAD 1,318.42 is within CAD 4.75
✓ Result survived itinerary selection

⚠ Retailer markets the flight as DE 2455 instead of WS 5943
⚠ Fare basis not exposed
⚠ Validating carrier not confirmed
```

Another example:

```text
MISMATCH — 42%

✓ Same physical flight
✓ Same departure date

✕ Retailer price is CAD 2,568.20
✕ Booking class appears to differ
✕ WestJet-marketed WS 5943 not reproduced
```

The scoring formula must be configurable and unit tested.

A suggested weighting:

* journey and segment identity: 30%;
* cabin identity: 15%;
* price identity: 20%;
* marketing/operating identity: 10%;
* fare identity: 15%;
* verification depth: 10%.

A hard mismatch should override the score for:

* wrong date;
* wrong origin or destination;
* economy instead of business;
* missing long-haul segment;
* passenger-count mismatch;
* price above the watch’s maximum.

# 12. Verification modes

Support several verification modes because not every website can be safely or reliably queried in the same manner.

## Mode A: Passive page verification

When I manually visit a supported booking site, the content adapter:

* detects the search;
* compares the visible results with active watches;
* records matching observations;
* updates the side panel;
* alerts me when the threshold is met.

This should be the most reliable mode.

## Mode B: User-initiated guided verification

From the watch, I click:

**Verify now**

FareProof:

1. opens the appropriate search page in a new tab;
2. pre-populates query parameters when supported;
3. waits for the page to load;
4. extracts visible results;
5. highlights likely matches;
6. optionally guides me through one additional click;
7. observes repricing;
8. records the deepest stage reached.

Do not automatically accept terms, sign in, solve CAPTCHAs, submit passenger forms, or click a final purchase button.

## Mode C: Periodic lightweight recheck

Use `chrome.alarms` to periodically evaluate due watches.

Because browser extensions and retailer pages may require an active browser session or manual interaction:

* do not promise 24/7 server-grade monitoring;
* mark each check as attempted, completed, blocked, throttled, or requiring interaction;
* do not create an endless stream of background tabs;
* limit concurrent verification;
* use exponential backoff;
* enforce domain-specific cooldowns;
* stop automatic checks after repeated blocking;
* expose the exact last-check status.

Where a site cannot be safely checked in the background, create a ready-to-run verification task and notify me:

**Fare watch is due — open verification**

## Mode D: Search-page opportunistic verification

Whenever I browse a supported airfare site, compare the current results against every relevant active watch.

For example, a YVR–FRA result should only be compared against watches with compatible:

* airports;
* travel dates;
* passenger count;
* trip type.

This passive intelligence should operate without requiring me to select a watch first.

# 14. BookWithMatrix integration

BookWithMatrix should be treated as a handoff and reconstruction channel, not automatically as the ultimate source of truth.

Support these workflows:

1. Read an ITA/BookWithMatrix JSON payload.
2. Generate or open the most precise BookWithMatrix handoff possible.
3. Capture any retailer links it produces.
4. Track which marketing carrier, operating carrier, booking class, fare basis, and price survive the handoff.
5. Open resulting retailers in controlled tabs.
6. compare each retailer’s result independently;
7. do not collapse all retailers into one observation.

A verification chain should look like:

```text
ITA Matrix
  → BookWithMatrix
    → Retailer A
    → Retailer B
    → Airline direct
```

Store each transition:

```ts
interface VerificationHop {
  fromSite: string;
  toSite: string;
  openedAt: string;
  sourceUrl: string;
  destinationUrl: string;
  preservedFields: string[];
  lostFields: string[];
  resultingObservationId?: string;

```

Display exactly where the price or itinerary diverged.

Example:

```text
ITA: CAD 1,313.67
BookWithMatrix handoff: CAD 1,313.67 metadata preserved
Retailer landing page: search reconstructed
Retailer results: CAD 2,586.22
Divergence: WS 5943 replaced with DE 2455
Fare basis DZ0D0HNS no longer present
```

# 15. Dynamic-page observation

Modern booking pages often update without full navigation.

Implement:

* `MutationObserver`;
* route-change detection;
* `history.pushState` and `history.replaceState` awareness where appropriate;
* debounced extraction;
* duplicate-observation suppression;
* page-stability detection;
* loading-state awareness;
* stale-result detection.

Do not extract continuously on every DOM mutation.

Use a pipeline such as:

```text

→ debounce
→ detect relevant region changed
→ wait for result stability
→ extract
→ normalize
→ hash
→ compare against last observation
→ publish only if materially different
```

# 19. Alert decision engine

An observation is notification-eligible only when:

```ts
priceWithinConfiguredTolerance
&& itineraryPassesHardConstraints
&& cabinPassesHardConstraints
&& comparisonScore >= minimumConfidence
&& verificationStage >= minimumStage
&& !isDuplicateAlert
```

Provide separate urgency levels:

* **Critical**: exact/cheaper price and deep verification;
* **High**: near price with strong itinerary match;
* **Medium**: search result reproduced but not selected;
* **Informational**: possible match requiring manual verification.

Allow me to configure which urgency levels trigger desktop notifications.

# 20. Scheduled checks and resource control

Implement a scheduler that respects browser and site constraints.

Requirements:

* one central alarm dispatcher;
* persist next-due times;
* avoid one Chrome alarm per observation;
* process only a small batch per cycle;
* maximum one active verification tab per domain;
* configurable global concurrency;
* configurable domain cooldown;
* random jitter;
* exponential backoff;
* pause when offline;
* retry transient network failures;
* do not retry authentication or CAPTCHA failures automatically;
* close extension-created tabs only when safe;
* never close tabs opened manually by the user;
* mark every tab with ownership metadata.

Suggested statuses:

```ts
type CheckStatus =
```

# 25. Watch state machine

Use an explicit state machine.

```text
Captured
  ↓
PendingVerification
  ↓
Checking
  ├─ NoMatch
  ├─ PossibleMatch
  ├─ StrongMatch
  ├─ ExactMatch
  ├─ Repriced
  ├─ ManualActionRequired
  ├─ Blocked
  └─ AdapterFailure
```

A watch can return to `Checking` later.

Keep:

* current state;
* best-ever match;
* latest match;
* lowest observed price;
* deepest verification stage;
* last successful verification;
* consecutive failures;
* next scheduled attempt.

Do not overwrite the best historical result when a later check fails.

# 28. False-positive protection

The extension must be conservative.

Never notify as a strong match when:

* the displayed cabin applies only to a short segment;
* the transatlantic segment is economy;
* the price is a monthly payment or deposit;
* the amount excludes mandatory taxes;
* the price is for a child or special passenger type;
* the price is per direction of a required round trip;
* the price is in another currency misread as CAD;
* the result date differs because of timezone parsing;
* the flight number belongs to a different day;
* the site shows “from” pricing;
* the fare is waitlisted;
* the result has already repriced;
* the retailer result is stale;
* the itinerary is mixed cabin and violates the watch rule.

Add explicit detectors for:

* “from”;
* “starting at”;
* “per month”;
* “excluding taxes”;
* “member price”;
* “round trip required”;
* “sold out”;
* “unavailable”;
* “price changed”;
* “mixed cabin.”

# 29. Currency handling

Do not compare formatted currency strings directly.

Normalize currency and minor units.

Keep:

* source currency;
* source amount;
* converted amount, if shown;
* conversion rate;
* rate timestamp;
* conversion source;
* whether conversion was performed by the retailer or by FareProof.

By default, alert only on original-currency comparisons.

If a retailer displays another currency, classify the price as provisional unless:

* the conversion source is known;
* the user enables cross-currency comparison;
* the conversion tolerance accounts for exchange-rate movement and card fees.

Never silently treat USD 1,313 as CAD 1,313.

# 39. Quality bar

Do not build a shallow proof of concept.

Avoid:

* giant files;
* untyped message payloads;
* brittle selectors scattered throughout the code;
* hidden global state;
* unpersisted service-worker assumptions;
* price-only matching;
* false “bookable” claims;
* uncontrolled tab creation;
* unlimited polling;
* broad host permissions;
* silent failures;
* hard-coded CAD-only logic;
* coupling the entire extension to ITA’s current DOM;
* storing sensitive browsing data unnecessarily.

Prefer:

* explicit contracts;
* typed schemas;
* adapter isolation;
* provenance;
* deterministic comparison;
* explainable decisions;
* conservative confidence;
* persistent state;
* graceful degradation;
* manual fallbacks;
* actionable diagnostics.
